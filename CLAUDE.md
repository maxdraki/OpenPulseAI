# OpenPulseAI — Project Context

## What this is

A local-first work journal and knowledge base. Automated collectors gather activity from GitHub, file systems, and other sources on schedules. An LLM synthesizes entries into curated wiki-style theme pages with cross-references. Users review and approve updates before they become permanent. Stakeholders can query the knowledge base via MCP without interrupting you.

## Architecture

pnpm workspace monorepo with 4 packages:

| Package | Purpose |
|---------|---------|
| `packages/core` | Vault I/O, config, LLM provider abstraction (Anthropic/OpenAI/Gemini/Ollama), skills system (loader/runner/scheduler/eligibility/security), orchestrator, logger, shared types |
| `packages/mcp-server` | MCP server (stdio + HTTPS) with 7 tools (`record_activity`, `ingest_document`, `query_memory`, `search_index`, `read_theme`, `chat_with_pulse`, `submit_update`), 1 resource (`openpulse://index`), and 2 prompts (`summarize_my_week`, `what_do_i_know_about`) |
| `packages/dream` | Dream Pipeline: multi-tag classify → wiki-style synthesize with [[cross-references]] → generate index.md → append log.md → batch pending review |
| `packages/ui` | Control Center web app (Vite + vanilla TS + custom CSS). Dev API server at `server.ts`. 11 pages: Dashboard, Chat, Review, Themes, Journals, Skills Evidence, Data Sources, Schedule, Logs, Settings, Help |

## Key patterns

- **MCP SDK**: `@modelcontextprotocol/sdk` v1.29.0. Server uses `McpServer` with `server.tool()` (4-arg overload). HTTPS transport (`packages/mcp-server/src/http.ts`) via `StreamableHTTPServerTransport` for Claude Desktop's custom-connector dialog — requires a bearer token (`packages/mcp-server/src/http-auth.ts`: auto-generated on first run, persisted at `~/OpenPulseAI/mcp-token` mode 0600) on every `/mcp` request, as `Authorization: Bearer` or `?token=` (the connector dialog only takes a URL, so the token is embedded there); `/health` stays open. stdio transport (the one-click Settings/Help flow) is unauthenticated by design — Claude Desktop spawns that process itself, so it's inherently trusted. Note: in stateless mode (`sessionIdGenerator: undefined`), the SDK requires a fresh `StreamableHTTPServerTransport` + server per request — a shared transport instance rejects the second request ("Stateless transport cannot be reused across requests"), per the SDK's own `examples/server/simpleStatelessStreamableHttp.ts`.
- **Outbound MCP (Connect Aigis)**: `packages/core/src/aigis/client.ts` — a per-call client (`Client` + `StreamableHTTPClientTransport`, connect → call → close, no long-lived state) to aigis.bio's remote MCP server, aigis.bio being the user's candidate-knowledge product (OpenPulseAI's second purpose is a proof-of-work journal for it). `testAigisConnection(config)` lists tools and never throws; `callAigisTool(config, name, args)` calls an arbitrary tool and distinguishes transport errors from tool-level errors via `transportError`. Both take a 30s default timeout (overridable, mainly for tests). Bearer auth via `requestInit.headers` when `authToken` is set. Settings UI (`packages/ui/src/pages/settings.ts`'s "Connect Aigis" card) and dev API (`GET`/`POST /api/aigis-config`, `POST /api/aigis-test` in `packages/ui/server.ts`) are the foundation only — connecting is review-gated plumbing; nothing auto-submits yet (later tasks add a scheduled rollup draft and submit-on-approval).
- **LLM calls**: All go through `LlmProvider` interface in `packages/core/src/llm/provider.ts`. Factory at `createProvider(config)`. Adapters for Anthropic, OpenAI, Gemini, Ollama. Temperature parameter supported on all calls. Every adapter wraps its raw call in the shared `withRetry()` helper (`packages/core/src/llm/retry.ts`): exponential backoff + jitter (base 1s, factor 2, max 4 retries, cap 30s), retries on 429/5xx/network errors, never retries 4xx client errors, honors `retry-after` when present. Each adapter also accumulates token/call/retry totals in a per-instance `UsageAccumulator` (`packages/core/src/llm/usage.ts`), exposed via optional `getUsageTotals()`/`resetUsage()` on `LlmProvider`. The Dream pipeline resets usage at the start of each run and logs per-run totals (`vaultLog("info", "Dream pipeline usage", ...)`); the Dashboard surfaces the latest run's totals via `GET /api/dream-usage`.
- **Skills**: SKILL.md files in `~/OpenPulseAI/skills/` (user) and `packages/core/builtin-skills/` (bundled). Frontmatter: name, description, schedule (cron), lookback, requires (bins, env), config (user-configurable fields with `{{placeholder}}` substitution). 12 bundled: github-activity, gitlab-activity, folder-watcher, google-daily-digest, obsidian-notes, confluence-activity, jira-activity, linear-activity, trello-activity, todoist-activity, slack-activity, sentry-activity.
- **Orchestrator**: `packages/core/src/orchestrator.ts`. Uses `croner` for cron scheduling. Barrier pattern: auto-triggers dream pipeline when all enabled collectors have run since last dream. JSON state file at `vault/orchestrator-state.json`. Heartbeat every 60s. Missed-run detection on startup. Also schedules compaction, schema-evolution, and Aigis-rollup pipelines (each with its own `<name>Pipeline` state section, default cron, and `trigger<Name>()`/`get<Name>PipelineState()` methods). The Aigis rollup pipeline (default weekly, Sunday 17:00) is gated on the host-provided `isAigisEnabled()` callback — it no-ops (schedule exists, every run is skipped) until aigis.bio is connected; it never needs the dream-lock guard compaction uses, since it only reads (never writes) hot/cold journals, the vault git log, and warm theme content.
- **Aigis rollup pipeline**: `packages/dream/src/aigis-rollup.ts` + `aigis-rollup-cli.ts` (mirrors `compact-cli.ts`'s pattern, incl. dream-lock acquisition and the realpath main-guard). On its schedule, drafts ONE candidate-relevant journal update pending (`theme: "aigis-rollup-<periodEnd-date>"`, sub-kind `aigisRollup: {periodStart, periodEnd, cadence}`) from the period's vault git history (`vaultLogSince` in `packages/core/src/vault-git.ts` — commit subjects + touched warm themes, `--since`-only, gracefully `[]` without git), hot/cold journal entries in the window, and the touched themes' current content (capped ~40k chars, prioritized by commit-touch frequency). Anti-hallucination framing mirrors `synthesize.ts`; first-person professional tone (represents the user to aigis.bio). A no-activity period writes nothing. Drafting a new rollup removes any existing pending `aigisRollup` update whose period OVERLAPS the new one (not an exact periodEnd-date match), replacing rather than stacking it. Never touches warm/ and never submits — that's a separate, approval-gated step.
- **Dream Pipeline**: Multi-tag classification (deterministic first: file paths, repo names, headings → LLM fallback for ambiguous entries). Synthesis with `[[wiki-link]]` cross-references. Auto-generates `vault/warm/index.md` and appends to `vault/warm/log.md`. Batch review with shared `batchId`. Per-theme failure isolation: one theme's synthesis failing (after the provider's own retries) is logged and skipped rather than aborting the batch (`packages/dream/src/synthesize.ts`); `runDreamPipeline` (`packages/dream/src/index.ts`) then applies a conservative ledger rule — an entry is only marked processed if *every* theme it was classified into succeeded, since one entry can carry multiple theme tags — and reports `failedThemes`/`deferredEntryCount` in the run result and log lines. Project/source-summary pages use append/patch synthesis instead of a whole-page rewrite once a page has grown past a small-page threshold (>2 sections and >1500 chars): the page is modeled as frontmatter/meta/preamble/`##`-sections (`packages/dream/src/page-patch.ts`'s `parsePageSections`/`serializeSections`, an exact-round-trip pair), the LLM is given an outline + only the likely-relevant sections and asked for a small JSON list of ops (`append_to_section`/`replace_section`/`add_section`/`update_meta`), and `applyPatch` deterministically applies them — untouched sections survive byte-identical, so output tokens scale with the delta, not the page size. Any ambiguity (unparsable ops, a rejected op — e.g. unknown heading or a `replace_section` that would shrink a section >50% — an empty ops list, or a truncated completion) falls back to the existing whole-page rewrite, which keeps its own truncation/shrinkage guards; concept/entity pages are unaffected (still two-pass fact-store synthesis). `synthesizeToPending`'s `onPatchOutcome` callback and `runDreamPipeline`'s `patchSynthesis: {patch, fallback}` result field report which path each eligible theme took. Fact-store hygiene (`packages/dream/src/facts.ts`): every fact gets a stable id (sha256-16 of normalized claim text + sourceId), so re-extracting the same fact on a later run is deduped on ingest rather than appended again (legacy lines without an `id` compute one on read). Facts are never deleted on conflict — pass-1 extraction may emit `supersedes: [<factId>]` referencing currently-active facts it contradicts/updates (guarded against self-supersession and unknown ids), which marks the old line `supersededBy`/`supersededAt` via an atomic tmp+rename rewrite (mirrors `ledger.ts`'s idiom) while keeping it in the file as history. Pass-2 resynthesis and concept compaction (`compact-cli.ts`'s `compactConcept`) both read ACTIVE facts only. Once a theme's fact file exceeds 300 lines or 100KB, `compactFactStore` archives superseded facts to `_facts/<theme>.archive.jsonl` (append-only) and rewrites the live file to active-only content — moved, never dropped.
- **Security**: Skill scanner (`scanSkillForThreats`) blocks non-builtin skills with high-severity findings (network exfiltration, destructive commands, credential access, privilege escalation). Shell-escaping on config values. Env var filtering strips secrets unless declared in `skill.requires.env`. The UI dev API server's bearer guard (`packages/ui/server.ts`) is always on, not opt-in: an explicit `OPENPULSE_API_TOKEN` wins, otherwise a persistent token is auto-generated at `~/OpenPulseAI/ui-token` (mode 0600, shared with `vite.config.ts` via `packages/ui/dev-token.ts` so the browser bundle gets it as `import.meta.env.VITE_OPENPULSE_TOKEN` — see `authHeaders()` in `src/lib/tauri-bridge.ts`).
- **Vault**: All data in `~/OpenPulseAI/vault/`. Hot = raw journals, Warm = curated theme pages + index.md + log.md, Pending = `warm/_pending/*.json`, Cold = monthly archives. Logs in `vault/logs/*.jsonl`. `vault/` is auto-adopted as a self-contained git repo (`packages/core/src/vault-git.ts`) — every approve (single or "Approve All" batch), theme merge/rename/delete, dream pipeline run, and lint/rebuild-meta write auto-commits with a structured message (`ensureVaultRepo`/`commitVault`, called from `Vault.init()`); degrades silently (warns once) if the `git` binary is missing. The Review UI's per-card "Diff" toggle (`packages/ui/src/lib/diff.ts`) renders a line-based before/after diff against a pending update's `previousContent`.
- **Retrieval**: Local hybrid search index over warm themes (`packages/core/src/search/`, barrel-exported from `@openpulse/core`) — SQLite FTS5 (`chunkTheme`/`index-db.ts`) fused via RRF with local embeddings when available (`embeddings.ts` degrades to FTS-only under vitest/missing models). `searchIndex(vault, query, opts)` never throws; `rebuildIndex`/`updateThemeInIndex`/`removeThemeFromIndex` keep it in sync. Every retrieval consumer is now index-backed: `query_memory` and `search_index` (MCP tools) return ranked chunks (theme/heading/snippet), not whole concatenated theme files; `chat_with_pulse` (`packages/mcp-server/src/tools/chat-with-pulse.ts`) fans a chat message out into one query per content keyword (`searchChatChunks` — a single AND'd full-sentence query would rarely match, since `sanitizeFtsQuery` ANDs every term) and assembles context under a documented `CONTEXT_CHAR_CAP` (~24k chars), swapping in a full theme page over fragments when a theme has multiple strong hits and it still fits; no matches (even after one rebuild+retry) falls back to the `index.md` page map alone rather than loading every theme. The Themes page search box (`GET /api/search`, debounced ~250ms) uses the same index. `packages/ui/src/lib/approve.ts` calls `updateThemeInIndex`/`removeThemeFromIndex` (fire-and-forget-safe, never fails the approve) after every write/merge/rename/delete so approved pages are searchable immediately; the UI dev server rebuilds the index in the background on startup if it's missing.
- **Config**: `~/OpenPulseAI/config.yaml` with llm provider/model/apiKey/baseUrl and an optional `aigis` section (`endpoint`, `authToken?`, `submitTool` default `aigis_submit_journal`, `enabled`) for the outbound connection to aigis.bio's remote MCP server — see `packages/core/src/config.ts`'s `isValidAigisEndpoint`/`parseAigisConfig` (a non-https endpoint forces `enabled: false`). Skills are filesystem-based with optional `vault/skill-config/<name>.json` for user settings.
- **UI**: No framework. Vanilla TS with custom CSS (Shoelace loaded for theme CSS only, not used as components). Hash-based routing. Use `confirmDialog()` from `src/lib/dialog.ts` instead of `window.confirm()`. Dev API server in `server.ts` bridges UI to vault filesystem and starts the orchestrator.
- **ESM**: All packages use `"type": "module"`. Imports use `.js` extensions.
- **Tauri**: v2 desktop wrapper in `src-tauri/`. Rust's ONLY job now is to keep the UI/API server sidecar (`openpulse-ui-server`, built from `packages/ui/server.ts`) alive and hand the webview its address — `src-tauri/src/server_sidecar.rs` spawns it once at startup (long-lived, via the shell plugin's `spawn()`/event-stream API, not the one-shot `.output()` the old `sidecar.rs` used), parses its `OPENPULSE_SERVER_READY port=<n>` stdout line, restarts it up to 3× with backoff if it dies unexpectedly, and sends SIGTERM (not `CommandChild::kill()`, which is SIGKILL) on real app exit (`RunEvent::ExitRequested`). Before spawning it probes `<vaultRoot>/ui-server.json` + a real HTTP request to avoid double-spawning over a dev-run `npx tsx server.ts`. The ONE Tauri command left is `get_server_info` → `{ port, token }` (port from readiness, token from `~/OpenPulseAI/ui-token`); `config.rs`/`skills.rs`/`discovery.rs` and the old `sidecar.rs` one-shot commands are gone, and `vault.rs` is trimmed to just the `AppState { vault_root }` the sidecar spawn needs (its old vault-I/O commands removed) — `packages/ui/src/lib/tauri-bridge.ts` now talks to the server directly over `fetch` in both runtimes (base URL fixed in the browser, resolved via `get_server_info` and cached in Tauri). `server.ts`'s `resolveBin()`/`resolveBuiltinSkillsDir()` read `OPENPULSE_BIN_DIR`/`OPENPULSE_BUILTIN_SKILLS_DIR` (set by the supervisor) so its dream/skills/aigis-rollup subprocess spawns and skills discovery work with the sidecar's cwd instead of a dev-only `process.cwd()`-relative guess. System tray + close-to-hide window lifecycle (`src-tauri/src/tray.rs`) unchanged: closing the main window hides it instead of exiting; tray menu offers "Open OpenPulseAI" / "Quit"; macOS toggles Dock visibility (`ActivationPolicy`) on hide/show — window hide/show never touches the sidecar. Not yet tested end-to-end with a real display server (see `.superpowers/sdd/task-20-report.md`'s manual verification steps).

## Build & test

```bash
pnpm install && pnpm build    # Build all packages
pnpm vitest run               # Run all tests (~3632 tests)
pnpm build:sea:mcp            # SEA binary for MCP server
pnpm build:sea:dream          # SEA binary for dream pipeline
pnpm build:sea:skills         # SEA binary for skills CLI
pnpm build:sea:ui             # UI API server bundled as a Tauri sidecar (src-tauri/sidecars/)
pnpm build:desktop            # Full Tauri desktop build
```

## Dev server

```bash
cd packages/ui
npx tsx server.ts &            # API server on :3001 (starts orchestrator)
npx vite --port 1420 &        # UI on :1420
```

`server.ts` exports `startServer(opts)` (port/vaultRoot/host/tokenPath overrides) and is also the always-on Tauri sidecar entry point — `npx tsx server.ts` and the bundled binary (`scripts/build-sidecar-ui.sh` → `src-tauri/sidecars/openpulse-ui-server-<triple>`) both boot through the same function, just sourcing options from different places (no args/env for dev; `OPENPULSE_PORT`/`OPENPULSE_VAULT`/`--port` for the sidecar). On EADDRINUSE it tries the next 10 ports; once listening it prints `OPENPULSE_SERVER_READY port=<port>` to stdout (parsed by the Rust supervisor) and writes `<vaultRoot>/ui-server.json` (`{port, pid, startedAt}`, mode 0600) as a fallback discovery file for the webview. SIGTERM/SIGINT stop the orchestrator, close the server, and remove the discovery file before exiting 0.

Both processes read the same auto-generated bearer token from `~/OpenPulseAI/ui-token` (`packages/ui/dev-token.ts`) — whichever starts first creates it, so there's no required startup order. `server.ts`'s `/api` guard is always on (no more "unset token → open").

Note: `_lint.md`'s two "missing concept page" signals are distinct — "concept candidates" are terms seen often in classification (raw entry-mention frequency, ≥3 entries, tracked in `_concept-candidates.json`), while "coverage gaps" are dangling `[[wiki-links]]` referenced by ≥3 *themes* with no page yet (computed fresh each lint run from the link graph, `COVERAGE_GAP_MIN_MENTIONS` in `lint-structural.ts`) — same threshold, different data source.
- Phase 4 Aigis bridge: outbound MCP connection to aigis.bio (Settings page "Connect Aigis" card — endpoint/token/enabled, gated on a valid https endpoint both at the runtime level and in the Settings/Schedule display so an invalid endpoint never shows as "enabled" while silently no-op'ing), a scheduled review-gated rollup pipeline (`packages/dream/src/aigis-rollup.ts`, orchestrator-scheduled, drafts a pending update for human approval — never auto-submits), and submit-on-approval (approving an `aigisRollup`-kind pending update posts it to aigis.bio via the configured submit tool, recorded in `vault/aigis/submissions.jsonl` with a Settings-page retry affordance for failed/skipped attempts).
